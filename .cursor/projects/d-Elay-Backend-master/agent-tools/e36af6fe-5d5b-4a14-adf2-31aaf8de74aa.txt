## Summary

Google launched **Antigravity 2.0** at I/O 2026 (May 19, 2026), shipping two distinct products — the **Antigravity IDE** (desktop app) and a new **Antigravity CLI** (`agy`, Go-based) — and announced that **Gemini CLI is being retired on June 18, 2026** for most users. Because the IDE and CLI use **separate global-scope config trees**, this issue proposes to **split rulesync's single `antigravity` target into `antigravity-ide` and `antigravity-cli`**, expand the feature coverage to match v2 (MCP, hooks, subagents, permissions, global scope), and define how rulesync handles the Gemini CLI deprecation.

> **Decision (updated):** Go with the **split** approach. The earlier "keep unified" lean was based on the assumption that the IDE and CLI share config; verification against the official docs (see the [verification comment](https://github.com/dyoshikawa/rulesync/issues/1679#issuecomment-4539312691)) showed that **project-scope files are shared but global-scope paths diverge** (`~/.gemini/antigravity/` for the IDE vs `~/.gemini/antigravity-cli/` for the CLI), which a single target cannot satisfy simultaneously.

## Motivation / Purpose

- rulesync's current `antigravity` target was built against the **November 2025** (v1) IDE and supports only **3 features**: rules, commands (workflows), skills — all project scope, no MCP/hooks/subagents/permissions (`src/types/tool-targets.ts`, `README.md:94`).
- Antigravity 2.0 ships **two products with distinct global config trees**, so one identifier cannot generate correct global-scope output for both.
- The Antigravity CLI **fully replaces Gemini CLI**, preserving Agent Skills, Hooks, Subagents, MCP, and Extensions (now "plugins") ([Google Developers Blog](https://developers.googleblog.com/an-important-update-transitioning-gemini-cli-to-antigravity-cli/), [CLI Migration doc](https://antigravity.google/docs/gcli-migration)). rulesync's `geminicli` target is its richest integration (8 features), so its users need a concrete `geminicli → antigravity-cli` migration path.

## Proposed Specification

### 1. Split `antigravity` → `antigravity-ide` + `antigravity-cli`

Add two new target identifiers and retire the single `antigravity` identifier:

- **`antigravity-ide`** — the desktop editor. Inherits the current `antigravity` behavior (IDE paths) and adds global scope.
- **`antigravity-cli`** — the `agy` CLI. New target covering the Gemini-CLI-class feature set.
- **`antigravity`** — keep as a **deprecated alias for `antigravity-ide`** (the current implementation already targets IDE paths: `.agent/...`, `~/.gemini/antigravity/skills/`), so existing `rulesync.jsonc` configs keep working. Emit a deprecation note pointing to the two new identifiers. (Final alias policy — see Open Questions.)

Both new identifiers must be registered in `src/types/tool-targets.ts` (`ALL_TOOL_TARGETS`) and wired into every feature processor's factory map plus `rules-processor.ts` conventions.

### 2. Per-target feature/scope plan

**`antigravity-ide`** (default output `.agents/`, back-compat read of `.agent/`):

| Feature | Project | Global | Path(s) |
| --- | :-: | :-: | --- |
| rules | ✅ | ✅ | `.agents/rules/`; global `~/.gemini/GEMINI.md` |
| commands (workflows) | ✅ | ✅ | `.agents/workflows/`; global `~/.gemini/antigravity/global_workflows/` |
| skills | ✅ | ✅ | `.agents/skills/`; global `~/.gemini/antigravity/skills/` |
| mcp | ✅ | ✅ | global `~/.gemini/antigravity/mcp_config.json` (workspace path TBD) |
| hooks | ✅ | ✅ | `.agents/hooks.json`; global `~/.gemini/config/hooks.json` |

**`antigravity-cli`** (default output `.agents/`):

| Feature | Project | Global | Path(s) |
| --- | :-: | :-: | --- |
| rules | ✅ | ✅ | `GEMINI.md`/`AGENTS.md` + `.agents/rules/`; global `~/.gemini/GEMINI.md` |
| skills | ✅ | ✅ | `.agents/skills/`; global `~/.gemini/antigravity-cli/skills/` |
| mcp | ✅ | ✅ | `.agents/mcp_config.json`; global `~/.gemini/antigravity-cli/mcp_config.json` |
| hooks | ✅ | ✅ | `.agents/hooks.json`; global `~/.gemini/config/hooks.json` |
| permissions | | ✅ | `~/.gemini/antigravity-cli/settings.json` (`permissions.allow`/`deny`) |
| subagents | ? | ? | standalone path unconfirmed (bundled under plugins `agents/`) — see Open Questions |

Note: project-scope outputs (`.agents/rules/`, `.agents/skills/`, `.agents/hooks.json`) are **identical** for both targets; selecting both simply writes the same files idempotently. The split exists to get **global scope** right.

### 3. Implementation checklist (per `.claude/rules/feature-change-guidelines.md`)

- New classes: `antigravity-ide-*` / `antigravity-cli-*` for each supported feature (reuse the existing antigravity rule/command/skill logic for the IDE; reuse `geminicli-mcp.ts` patterns for MCP with `serverUrl`/`disabledTools`).
- A **new Antigravity hooks generator** in the `hooks.json` (Claude-Code-like) shape — do **not** reuse `geminicli-hooks.ts`.
- Default the directory output to `.agents/` (plural); continue reading `.agent/` (singular) on import for back-compat.
- Update `rules-processor.ts` `additionalConventions`, review frontmatter precedence, run `pnpm dev gitignore` to refresh `.gitignore`, and implement project+global scope where the product supports it.
- Keep README/`docs/**` "Supported Tools and Features" + "Each File Format" in sync, and **preserve/extend the e2e happy-path Tool × Feature matrix** for both new targets.

### 4. Gemini CLI deprecation handling

- **Do not remove `geminicli`.** Enterprise access continues past June 18, 2026, countless existing `GEMINI.md`/`.gemini/` repos depend on it, and CLAUDE.md requires preserving the e2e matrix. Removal would be breaking.
- **Mark `geminicli` deprecated in docs** (README matrix + a note with the June 18, 2026 date and the enterprise exception) and point users to **`antigravity-cli`** as the successor.
- Once `antigravity-cli` covers rules/skills/mcp/hooks/subagents/permissions, document a **`geminicli → antigravity-cli` migration** in `docs/` (mirrors Google's own `agy plugin import gemini`).

### Acceptance criteria

- `antigravity-ide` and `antigravity-cli` registered and wired through all feature processors; `antigravity` kept as a deprecated alias.
- Feature/scope coverage implemented per the tables above, with global-scope paths correct per product.
- README/`docs/**` updated for the split and the `geminicli` deprecation note.
- e2e happy-path tests cover the new Tool × Feature cells for both targets.

## Open Questions

1. **`antigravity` alias policy** — Keep `antigravity` as a permanent deprecated alias for `antigravity-ide`, or schedule its removal in a future major? Should selecting `antigravity` emit a warning?
2. **CLI subagents** — Is there a standalone, file-based subagents config for `antigravity-cli` outside the plugin bundle (`~/.gemini/antigravity-cli/plugins/ /agents/`)? If subagents are only definable via plugins or the `define_subagent` tool, rulesync may not be able to generate them directly.
3. **Plugins** — "Plugins" are a bundle (skills + agents + rules + mcp + hooks). This doesn't fit rulesync's per-feature model; treat as out of scope, or add a dedicated bundle feature later?
4. **IDE permissions / global MCP workspace path** — The IDE manages permissions via UI (no obvious file to generate). Confirm the IDE's *workspace* MCP path (the docs only clearly state the global `~/.gemini/antigravity/mcp_config.json`).
5. **`.agents/` default migration** — Confirm flipping the default output from `.agent/` (singular) to `.agents/` (plural) for rules/workflows/skills, while keeping `.agent/` readable on import.

> ### ⚠️ Critical for the unify-vs-split decision (Open Question #1): global-scope paths DIFFER between CLI and IDE
> 
> The earlier body said CLI and IDE "share config." That is only true at **project/workspace scope**. The [CLI Migration doc](https://antigravity.google/docs/gcli-migration) and [CLI Features doc](https://antigravity.google/docs/cli-features) show that **global-scope paths diverge**:

> | Global skills | `~/.gemini/antigravity/skills/` | `~/.gemini/antigravity-cli/skills/` | + inherited `~/.gemini/skills/` |

> - **(A) Keep one `antigravity` target**, and make the *global*-scope paths configurable per-feature (e.g. a `variant: "ide" | "cli"` option), defaulting to one. Pro: small `--targets` surface; Con: a single global generate can't satisfy both products at once.
