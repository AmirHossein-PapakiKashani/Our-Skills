## Git lfs connection to Azure DevOps Server with ntlm causes HTTP Error 401. The requested resource requires user authentication

- Author: [@IS-GYG](https://github.com/IS-GYG)
- State: closed (completed)

**Describe the bug**
Have a repo that has binary files being tracked by git lfs.
Pushing binary works. But any form of pull causes the following error form the server

HTTP Error 401. The requested resource requires user authentication.

Git Lfs logs it as

batch response: No credentials are available in the security package

**To Reproduce**
Try pulling from a and Azure DevOps server with binary being tracked by git lfs :

git pull

or

git lfs fetch

**Expected behavior**
All binaries are fetched by git lfs

**System environment**
Client - Windows 10
Server- Azure Dev Ops
git-lfs/2.11.0 (GitHub; windows amd64; go 1.14.2; git 48b28d97)

> This looks like an NTLM problem of some sort. I believe that message means that the NTLM code can't find the credentials, but I don't know why that happens. You didn't include the version of Git LFS in your bug report, but it may be worth trying the latest version to see if it's fixed if you haven't already.
> 
> If not, then we'd need someone with access to a Windows system using NTLM to look into this, since none of the core team routinely uses Azure DevOps, NTLM, or Windows. You can also try using Kerberos support, which should be available in recent versions, instead of NTLM and see if that works better; it's also far more secure than NTLM.

**@IS-GYG** commented · Oct 8, 2020 at 5:13pm · Author

> @bk2204 , We have finally been able to resolve the issue. After all these years it turns out that Kerberos wasn't properly configure for our Azure DevOps instance. We finally insure that Kerberos was full working (and initially disable NTLM). Once we confirmed that Kerberos was full working, we had to re-enable NTLM (as a fallback) so that Git LFS can work. From what I understand, the Git LFS portion of AzureDevOps uses NTLM (or basic authentication), for it's Git LFS communication. But once Kerberos is properly configured NTLM will work with SSPI. The catch here is that it means you have to be logged into your Active Directory account (my issue was that I was logged into my normal Microsoft Account, but the credential manager store my Active Directory credentials), on windows. This isn't a major issue for our core devs (who are already logged into their AD account on the windows machine), but can be an issue for our remote contractors (will have to mandate that they login with an AD account and have it create a profile on their pc).
> 
> In short, there issue wasn't 100% with Git LFS client (although this wont have worked in previous version of GIT LFS when NTLM wasn't supported).
> 
> In conclusion, Git LFS still needs NTLM support (because for now Azure DevOPS Git LFS uses NTLM even if the Git part of it uses Kerberos, this is something will have to bring up with Microsoft).
> 
> Will go ahead and close this issue. On the other hand I am worried that #4247 might cause a regression if we upgrade (we are on version 2.11.0).
> 
> Thanks again for your support all these times
